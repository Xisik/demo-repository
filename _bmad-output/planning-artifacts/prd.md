# Product Requirements Document - bichcheongmo

**Author:** (๑´ڡ`๑)
**Date:** 2026-01-12

## Executive Summary

**Vision:** 성소수자 공조 단체 bichcheongmo의 활동 공유 페이지를 정적 페이지에서 동적 콘텐츠 관리 시스템으로 전환하여, 단체의 실제 활동을 지속적으로 증명하고 신뢰를 형성한다.

**Product Differentiator:** 백엔드 없이 노션(Notion)을 콘텐츠 관리 도구로 사용하여 운영자가 개발 지식 없이도 활동을 업데이트할 수 있는 기술적으로 단순한 방식으로 지속 가능한 운영을 보장한다.

**Target Users:**
- **Primary:** 성소수자 이슈에 관심이 있거나 관련 정보를 찾는 외부 방문자
- **Secondary:** 단체 운영자/콘텐츠 관리자 (기술 지식 없는 비개발자)

**Core Value Proposition:** "이 단체는 지금도 실제로 움직이고 있다"는 신뢰를 기술적으로 단순한 방식으로 지속 증명한다.

**Project Context:** Brownfield 프로젝트 - 기존 정적 웹사이트에 동적 기능 추가

## Success Criteria

### User Success

**사용자에게 "성공"이란:**

- 사용자가 단체의 실제 활동이 지금도 이어지고 있음을 명확히 인지하는 것
- 단체가 추상적인 이름이 아니라 구체적인 행동을 하는 집단으로 느껴지는 것

**사용자가 느끼는 감정 / 상태:**

- 신뢰감 ("아, 이 단체는 실제로 움직이고 있구나")
- 연대감 ("나와 같은 사람들을 위해 이런 활동을 하고 있네")
- 안심과 희망 ("혼자가 아니구나")

**사용자가 완료하길 원하는 주요 행동:**

- 최근 활동 목록을 훑어본다
- 관심 있는 활동의 상세 내용을 확인한다
- (선택적으로) 외부 채널(SNS, 문의 페이지 등)로 이어진다

### Business Success

**이 기능이 달성해야 하는 목표:**

- 단체의 신뢰도와 가시성 강화
- 외부 방문자에게 단체의 활동 실체를 증명
- 내부적으로는 활동 기록을 정리된 형태로 축적

**성공 판단 지표:**

**3개월 후:**
- 활동 공유 페이지 월간 조회수 증가
- 게시된 활동 수가 지속적으로 유지/증가
- 특정 활동 상세 페이지 조회 발생

**12개월 후:**
- 활동 페이지를 통해 유입된 문의 또는 참여 연결 사례
- 활동 기록이 단체의 연간 활동 보고 자료로 활용됨

**단체에 가져다주는 가치:**

- "말하는 단체"가 아니라 "행동하는 단체"라는 인식 형성
- 외부 협력, 연대 요청 시 신뢰 근거 자료로 활용 가능

### Technical Success

**백엔드 없이 운영 가능한 상태에서의 "성공":**

- 활동 콘텐츠를 개발 지식 없이도 업데이트 가능
- 데이터 소스 변경 없이도 사이트 재배포만으로 반영 가능
- GitHub Pages 환경에서 안정적으로 동작

**기술적 기준:**

- **성능:** 정적 페이지 수준의 빠른 로딩 유지
- **안정성:** 외부 데이터 소스 장애 시에도 페이지가 깨지지 않음
- **유지보수성:** 활동 추가/수정 시 코드 수정 최소화

### Measurable Outcomes

- 활동 공유 페이지 조회수
- 게시된 활동 수
- 활동 상세 페이지 평균 체류 시간
- 외부 링크(SNS, 문의 페이지) 클릭 수
- 활동 업데이트 주기(정기성)

## Product Scope

### MVP - Minimum Viable Product

**초기 필수 기능:**

- 활동 목록 표시
- 활동 기본 정보(제목, 날짜, 간단 설명)
- 활동 상세 내용 확인
- 외부 데이터 소스 기반 콘텐츠 로딩

### Growth Features (Post-MVP)

**이후 확장:**

- 활동 카테고리 분류
- 연도/월별 필터링
- 활동별 이미지 또는 첨부 자료
- 활동 공유(SNS 링크)

### Vision (Future)

**장기 비전:**

- 단체의 활동 기록이 역사 아카이브처럼 축적
- 활동 페이지가 연대 단체·개인에게 참여의 출발점이 됨
- 기술적으로는 필요 시 백엔드 또는 CMS로 자연스럽게 확장 가능

## User Journeys

### Journey 1: 외부 방문자 - 신뢰 형성 여정

**페르소나:** 김민수 (가명), 28세, 성소수자 이슈에 관심이 있는 사람

**Opening Scene - 어디서 만나는가:**

김민수는 인터넷에서 성소수자 관련 정보를 검색하다가 "bichcheongmo"라는 단체 이름을 처음 접했습니다. SNS에서 누군가 공유한 링크를 클릭하거나, 검색 엔진을 통해 단체 사이트에 유입되었습니다. 그는 호기심과 함께 약간의 불안감을 느끼고 있습니다. "이 단체가 정말 믿을 만한 곳일까? 단순히 이름만 있는 단체는 아닐까?"라는 의문이 들었습니다.

**Rising Action - 탐색의 시작:**

활동 공유 페이지에 진입한 김민수는 최근 활동 목록을 빠르게 훑어봅니다. 페이지가 빠르게 로드되고, 최근 날짜의 활동들이 시간순으로 정렬되어 보입니다. 각 활동에는 제목, 날짜, 간단한 설명이 명확하게 표시되어 있습니다. 그는 "아, 최근에도 활동을 하고 있구나"라는 생각을 하며 관심이 생깁니다.

**Climax - 신뢰의 순간:**

관심 있는 활동 하나를 클릭해 상세 내용을 확인합니다. 구체적인 활동 내용, 참여 인원, 실제로 일어난 일들이 상세히 기록되어 있습니다. 과장된 문구가 아니라 실제 기록을 보면서, 김민수는 "아, 이 단체는 실제로 움직이고 있구나"라는 확신을 갖게 됩니다. 단체가 추상적인 이름이 아니라 구체적인 행동을 하는 집단이라는 것을 인지합니다.

**Resolution - 새로운 현실:**

신뢰감과 연대감을 느낀 김민수는 "나와 같은 사람들을 위해 이런 활동을 하고 있네", "혼자가 아니구나"라는 안심과 희망을 경험합니다. 필요하다면 다른 페이지(소개, 문의, SNS)로 자연스럽게 이동하여 더 많은 정보를 얻거나 연락을 취할 수 있습니다. 이 흐름이 막힘 없이 자연스럽게 이어지면서, 김민수는 단체에 대한 긍정적인 인상을 갖게 됩니다.

**감정적 여정:**
- 시작: 호기심 + 불안감 ("믿을 만한 곳일까?")
- 탐색: 관심 증가 ("최근에도 활동을 하고 있구나")
- 클라이맥스: 신뢰 형성 ("실제로 움직이고 있구나")
- 해결: 안심과 희망 ("혼자가 아니구나")

**실패 시나리오 및 복구:**

- 활동 목록이 로드되지 않는 경우: 에러 메시지 표시, 기본 정적 콘텐츠로 폴백
- 외부 데이터 소스 장애: 캐시된 데이터 또는 기본 메시지 표시, 페이지가 완전히 깨지지 않음
- 활동이 너무 많아 탐색이 어려운 경우: (Growth 기능) 필터링 또는 검색 기능 활용

### Journey 2: 단체 운영자/콘텐츠 관리자 - 활동 업데이트 여정

**페르소나:** 박지영 (가명), 단체 운영진, 기술 지식이 없는 비개발자

**Opening Scene - 어디서 만나는가:**

박지영은 단체의 새로운 활동이 끝난 후, 이를 웹사이트에 반영해야 합니다. 그녀는 HTML, JavaScript, GitHub에 대한 지식이 없지만, 활동 기록을 정리된 형태로 축적하고 외부 방문자에게 단체의 활동 실체를 보여주고 싶어 합니다. 과거에는 개발자에게 의존해야 했지만, 이제는 스스로 업데이트할 수 있는 방법을 원합니다.

**Rising Action - 콘텐츠 작성:**

박지영은 노션(Notion)을 열고 활동별로 정해진 템플릿에 맞춰 콘텐츠를 작성합니다. 템플릿에는 제목, 날짜, 간단 설명, 상세 내용 필드가 명확하게 정의되어 있어 혼란 없이 작성할 수 있습니다. 그녀는 HTML이나 코드를 건드리지 않고, 단순히 노션 문서만 수정합니다.

**Climax - 자동 반영의 순간:**

노션에서 활동 콘텐츠를 작성하고 저장하면, 외부 데이터 소스로 연결된 정적 웹사이트에 자동으로 반영됩니다. (구현 방식은 Architecture 단계에서 결정되지만) 사이트 재배포 또는 자동 반영 방식으로 활동이 업데이트됩니다. 박지영은 개발자 도움 없이도 활동을 추가/수정할 수 있게 되었습니다.

**Resolution - 지속 가능한 운영:**

박지영은 실수를 해도 사이트가 깨지지 않는다는 것을 경험합니다. 콘텐츠 관리가 부담이 되지 않아 지속적으로 활동을 업데이트할 수 있게 되었습니다. 단체의 활동 기록이 정리된 형태로 축적되고, 외부 방문자에게 실시간에 가까운 활동 정보를 제공할 수 있게 되었습니다.

**감정적 여정:**
- 시작: 부담감 ("개발자에게 의존해야 하나?")
- 작성: 편의성 ("템플릿이 있어서 쉽네")
- 클라이맥스: 자율성 ("스스로 할 수 있다!")
- 해결: 지속 가능성 ("부담 없이 계속 업데이트 가능")

**실패 시나리오 및 복구:**

- 노션 템플릿 형식 오류: 명확한 에러 메시지와 수정 가이드 제공
- 데이터 동기화 실패: 재시도 메커니즘, 수동 동기화 옵션
- 잘못된 콘텐츠 입력: 유효성 검사, 미리보기 기능으로 확인 후 반영

3. **네비게이션 연계**
   - 다른 페이지(소개, 문의, SNS)로 자연스러운 이동

   - 노션을 콘텐츠 관리 도구로 사용
   - 템플릿 기반 콘텐츠 작성 지원

2. **자동 동기화 기능**
   - 노션 콘텐츠 변경 시 자동 반영
   - 사이트 재배포 또는 자동 동기화 방식
   - 개발 지식 없이도 업데이트 가능

3. **안정성 및 유효성 검사**
   - 실수해도 사이트가 깨지지 않음
   - 템플릿 형식 오류 시 명확한 에러 메시지
   - 미리보기 기능 (선택적)

4. **지속 가능한 운영**
   - 콘텐츠 관리 부담 최소화
   - 정기적인 활동 업데이트 지원
   - 활동 기록의 체계적 축적

## Web App Specific Requirements

### Project-Type Overview

이 프로젝트는 정적 웹사이트(Static Web Site) 구조를 유지하면서, 활동 공유 페이지에만 동적 렌더링 기능을 추가하는 하이브리드 접근 방식을 사용합니다. 기존 MPA(Multi-Page Application) 구조를 유지하여 SEO와 성능을 최적화합니다.

### Technical Architecture Considerations

**SPA vs MPA:**
- **MPA 유지**: 기존 정적 사이트 구조 유지
- **동적 렌더링**: 활동 페이지에만 동적 콘텐츠 로딩 추가
- **하이브리드 접근**: 정적 페이지의 장점(SEO, 성능)과 동적 콘텐츠의 유연성 결합

**데이터 소스 연동:**
- 노션(Notion)을 콘텐츠 관리 도구로 사용
- 노션 데이터를 외부 데이터 소스로 활용
- 정적 생성 또는 클라이언트 사이드 렌더링 방식 고려

### Browser Support Matrix

**Desktop 브라우저:**
- Chrome: 최신 2버전
- Edge: 최신 2버전
- Firefox: 최신 2버전
- Safari: 최신 2버전

**Mobile 브라우저:**
- iOS Safari: 15+ (권장)
- Android Chrome: 최신 2버전

**지원 범위:**
- 모던 브라우저 기능 활용 (CSS 변수, ES6+)
- 폴백 메커니즘 제공 (필요 시)
- 반응형 디자인 필수

### Responsive Design

**요구사항:**
- 모바일/태블릿/데스크톱 지원
- 터치 타겟 크기 준수 (최소 44x44px)

- 또는 서버 사이드 렌더링(SSR) 또는 클라이언트 사이드 렌더링(CSR) + SEO 최적화
- 각 활동에 고유 URL 및 메타데이터 제공

- 점진적 개선(Progressive Enhancement) 원칙 적용

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** 문제 해결 MVP + 경험 MVP 하이브리드

이 프로젝트는 두 가지 MVP 접근을 결합합니다:
- **문제 해결 MVP**: 단체의 활동 실체를 증명하는 핵심 문제 해결
- **경험 MVP**: 사용자가 "이 단체는 실제로 움직이고 있다"는 신뢰를 느끼는 핵심 경험 제공

**핵심 원칙:**
- 최소 기능으로도 사용자에게 가치 있는 경험 제공
- 기술적으로 단순하지만 신뢰를 형성하는 데 충분한 기능
- 운영자가 개발 지식 없이도 콘텐츠를 관리할 수 있는 구조

**Resource Requirements:**
- 개발자 1명 (풀타임 또는 파트타임)
- 운영자 1명 (콘텐츠 관리, 기술 지식 불필요)
- GitHub Pages 무료 호스팅
- 노션 무료 플랜 또는 유료 플랜 (API 접근 필요 시)

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**

1. **외부 방문자 - 신뢰 형성 여정**
   - 활동 목록을 빠르게 훑어볼 수 있음
   - 관심 있는 활동의 상세 내용을 확인할 수 있음
   - 단체가 실제로 활동 중임을 인지할 수 있음

2. **단체 운영자 - 활동 업데이트 여정**
   - 노션에서 활동 콘텐츠 작성 및 수정
   - 개발 지식 없이도 콘텐츠 업데이트 가능
   - 자동 동기화로 사이트에 반영

**Must-Have Capabilities:**

1. **활동 목록 표시**
   - 최근 활동 목록을 시간순으로 표시
   - 제목, 날짜, 간단 설명 포함
   - 빠른 로딩 속도 (정적 페이지 수준)

2. **활동 상세 내용 확인**
   - 클릭 시 상세 내용 표시
   - 구체적이고 현실적인 활동 기록
   - 과장 없는 실제 기록 중심

3. **외부 데이터 소스 기반 콘텐츠 로딩**
   - 노션을 콘텐츠 관리 도구로 사용
   - 노션 데이터를 외부 데이터 소스로 활용
   - 자동 동기화 (GitHub Actions, 5~30분 주기)

4. **기본 SEO 지원**
   - 활동 콘텐츠 검색 엔진 노출
   - 메타 태그 최적화
   - 구조화된 데이터 마크업

5. **기본 접근성**
   - WCAG 2.1 A 수준 준수
   - 모바일 반응형 디자인
   - 키보드 네비게이션 지원

**MVP에서 제외되는 기능 (Post-MVP):**
- 활동 카테고리 분류
- 연도/월별 필터링
- 활동별 이미지 또는 첨부 자료
- 활동 공유(SNS 링크)
- 실시간 동기화 (초기에는 주기적 업데이트)

### Post-MVP Features

**Phase 2 (Growth) - Post-MVP:**

1. **활동 카테고리 분류**
   - 활동 유형별 분류 (예: 상담, 교육, 행사 등)
   - 카테고리별 필터링 기능

2. **연도/월별 필터링**
   - 시간대별 활동 탐색
   - 검색 기능 (선택적)

3. **활동별 이미지 또는 첨부 자료**
   - 활동 사진 표시
   - 관련 자료 링크

4. **활동 공유 기능**
   - SNS 공유 링크
   - 개별 활동 URL 공유

5. **향상된 SEO**
   - 이미지 최적화
   - 고급 구조화된 데이터
   - 사이트맵 자동 생성

**Phase 3 (Expansion) - Vision:**

1. **활동 기록 아카이브**
   - 단체의 활동 기록이 역사 아카이브처럼 축적
   - 장기 보관 및 검색 기능

2. **참여 연결 기능**
   - 활동 페이지가 연대 단체·개인에게 참여의 출발점이 됨
   - 참여 신청 또는 문의 연결

3. **실시간 동기화**
   - 서버리스 프록시를 통한 거의 실시간 동기화
   - Webhook 기반 즉시 업데이트

4. **고급 접근성**
   - WCAG 2.1 AA 수준 달성
   - 스크린 리더 최적화
   - 다국어 지원 (선택적)

5. **확장 가능한 아키텍처**
   - 필요 시 백엔드 또는 CMS로 자연스럽게 확장
   - API 기반 구조로 전환 가능

- **폴백**: 페이지가 완전히 깨지지 않도록 정적 폴백 콘텐츠 제공

- **검증**: 활동 업데이트 주기 모니터링

**Resource Risks:**

**리스크 1: 개발 리소스 부족**
- **완화 전략**: MVP 범위를 최소화, 단순한 기술 스택 선택
- **대안**: 단계적 구현, 핵심 기능부터 우선 구현

**리스크 2: 운영자 부담 증가**
- **완화 전략**: 콘텐츠 관리 프로세스 단순화, 템플릿 제공
- **대안**: 초기에는 개발자가 지원, 점진적으로 운영자 자율성 확대

**리스크 3: 기술 부채 누적**
- **완화 전략**: 확장 가능한 구조 설계, 코드 품질 유지
- **대안**: 정기적인 리팩토링 계획, 점진적 개선

## Functional Requirements

### 활동 콘텐츠 표시 및 탐색

**FR1:** 외부 방문자는 활동 목록을 시간순으로 볼 수 있다.

**FR2:** 외부 방문자는 각 활동의 제목, 날짜, 간단 설명을 볼 수 있다.

**FR3:** 외부 방문자는 관심 있는 활동의 상세 내용을 확인할 수 있다.

**FR4:** 외부 방문자는 활동 상세 페이지를 정적 HTML로도 접근할 수 있다. (검색 엔진 노출을 위해 정적 생성 또는 프리렌더링 지원)

### 콘텐츠 관리 및 동기화

**FR5:** 운영자는 노션에서 활동 콘텐츠를 작성하고 수정할 수 있다.

**FR6:** 시스템은 노션의 활동 콘텐츠를 자동으로 동기화할 수 있다.

**FR7:** 운영자는 개발 지식 없이도 활동 콘텐츠를 업데이트할 수 있다.

**FR8:** 노션 수정 후 5~30분 이내에 사이트에 반영된다. (초기 MVP는 주기적 동기화, 향후 서버리스 프록시로 거의 실시간 확장 가능)

**FR9:** 동기화 실패 시 사용자에게 오류 메시지와 마지막 업데이트 시각을 표시하며, 페이지는 깨지지 않는다.

**FR10:** 운영자는 노션 템플릿에 따라 필수 필드(제목, 날짜, 요약, 본문, 슬러그 또는 ID, 공개 여부)를 작성할 수 있으며, 필수 필드가 없을 경우 시스템이 적절히 처리한다.

### 검색 엔진 최적화

**FR11:** 활동 콘텐츠는 검색 엔진에 노출될 수 있다.

**FR12:** 각 활동은 고유한 URL을 가진다.

**FR13:** 각 활동 상세 페이지는 메타 태그(title, description)와 Open Graph 태그(OG:title, OG:description, OG:image 등)를 제공한다.

### 접근성 및 사용성

**FR14:** 사용자는 키보드만으로 모든 기능에 접근할 수 있다.

**FR15:** 사용자는 모바일 기기에서 활동 페이지를 사용할 수 있다.

**FR16:** 사용자는 스크린 리더를 통해 활동 콘텐츠에 접근할 수 있다.

### 에러 처리 및 안정성

**FR17:** 외부 데이터 소스(노션) 장애 시에도 페이지가 완전히 깨지지 않고, 캐시된 데이터 또는 기본 메시지를 표시한다.

**FR18:** 시스템은 데이터 로드 실패 시 사용자에게 명확한 에러 메시지를 제공한다.

### 네비게이션 및 연계

**FR19:** 사용자는 활동 페이지에서 다른 페이지(소개, 문의, SNS)로 자연스럽게 이동할 수 있다.

**FR20:** 사용자는 활동 상세 페이지의 외부 링크(SNS, 문의 페이지 등)를 클릭할 수 있다.

### 데이터 구조 및 필드

**FR21:** 시스템은 활동 데이터의 필수 필드(제목, 날짜, 요약, 본문)를 인식하고 처리할 수 있다.

**FR22:** 시스템은 선택적 필드(이미지, 첨부 자료, 카테고리 등)를 지원할 수 있다.

**FR23:** 시스템은 활동의 공개 여부 상태를 처리할 수 있다. (공개/비공개)

### 성능 및 사용자 경험

**FR24:** 활동 목록 페이지는 빠르게 로드된다. (정적 페이지 수준의 성능)

**FR25:** 활동 상세 페이지는 빠르게 로드된다.

**FR26:** 사용자는 활동 목록을 스크롤하여 더 많은 활동을 볼 수 있다. (필요 시 페이지네이션 또는 무한 스크롤)

### 콘텐츠 표시 형식

**FR27:** 시스템은 활동의 날짜를 사용자가 이해하기 쉬운 형식으로 표시할 수 있다.

**FR28:** 시스템은 활동의 본문 내용을 읽기 쉬운 형식으로 표시할 수 있다. (마크다운 또는 HTML 렌더링)

**FR29:** 시스템은 활동 목록에서 최신 활동을 우선적으로 표시할 수 있다.

### 운영자 경험

**FR30:** 운영자는 노션에서 활동을 추가한 후 사이트에 자동으로 반영되는 것을 확인할 수 있다.

**FR31:** 운영자는 노션 템플릿을 사용하여 일관된 형식으로 활동을 작성할 수 있다.

**FR32:** 운영자는 실수로 잘못된 형식의 데이터를 입력해도 사이트가 깨지지 않는다.

## Non-Functional Requirements

### Performance

**NFR1:** Mobile(4G) 네트워크 환경에서 First Contentful Paint (FCP)는 1.5초 이하여야 한다.

**NFR2:** Mobile(4G) 네트워크 환경에서 Largest Contentful Paint (LCP)는 2.5초 이하여야 한다.

**NFR3:** Mobile(4G) 네트워크 환경에서 Time to Interactive (TTI)는 3.5초 이하여야 한다.

**NFR4:** 활동 목록 데이터 렌더링이 완료되는 시간은 캐시 미스 시 2.0초 이하여야 한다.

**NFR5:** 활동 목록 데이터 렌더링이 완료되는 시간은 캐시 히트 시 1.0초 이하여야 한다.

### Synchronization

**NFR6:** 노션에서 활동 콘텐츠 수정 후 사이트에 반영되는 동기화 지연은 P95 기준 30분 이하여야 한다. (동기화 주기: 10~30분)

**NFR7:** 동기화 실패 시 이전 캐시된 데이터를 유지하여 사용자에게 표시해야 한다.

### Reliability

**NFR8:** 외부 데이터 소스(노션) 장애 시에도 페이지가 완전히 깨지지 않아야 한다.

**NFR9:** 데이터 로드 실패 시 사용자에게 오류 메시지와 마지막 업데이트 시각을 표시해야 한다.

**NFR10:** 가능한 경우 캐시된 데이터를 폴백으로 사용하여 콘텐츠를 표시해야 한다.

### Search Engine Optimization (SEO)

**NFR11:** 각 활동은 고유한 URL을 가져야 한다.

**NFR12:** 각 활동 상세 페이지는 메타 태그(title, description)와 Open Graph 태그(OG:title, OG:description, OG:image 등)를 포함해야 한다.

**NFR13:** 가능한 경우 활동 상세 페이지는 정적 HTML로도 접근 가능하도록 배포 시 생성되거나 프리렌더링되어야 한다. (검색 엔진 크롤링을 위해)

### Accessibility

**NFR14:** 웹사이트는 WCAG 2.1 AA 수준을 목표로 하며, 최소한 A 수준을 준수해야 한다.

**NFR15:** 모든 기능은 키보드 네비게이션과 포커스 관리로 접근 가능해야 한다.

**NFR16:** 320px 너비의 모바일 화면에서도 모든 기능이 사용 가능해야 한다.

### Security

**NFR17:** 시스템은 민감한 데이터를 처리하지 않는다. (공개 활동 기록만 다룸)

**NFR18:** 노션 API 키 또는 토큰은 클라이언트 사이드 코드에 포함되지 않아야 한다. 서버 사이드 또는 GitHub Actions에서만 사용되어야 한다.

### Scalability

**NFR19:** 시스템은 일일 50~500명의 기본 사용자를 지원할 수 있어야 한다.

**NFR20:** 이벤트 시 일일 2,000~10,000명의 스파이크 트래픽을 캐싱 전략으로 대응할 수 있어야 한다.
