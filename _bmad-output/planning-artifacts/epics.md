# bichcheongmo - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for bichcheongmo, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 외부 방문자는 활동 목록을 시간순으로 볼 수 있다.

FR2: 외부 방문자는 각 활동의 제목, 날짜, 간단 설명을 볼 수 있다.

FR3: 외부 방문자는 관심 있는 활동의 상세 내용을 확인할 수 있다.

FR4: 외부 방문자는 활동 상세 페이지를 정적 HTML로도 접근할 수 있다. (검색 엔진 노출을 위해 정적 생성 또는 프리렌더링 지원)

FR5: 운영자는 노션에서 활동 콘텐츠를 작성하고 수정할 수 있다.

FR6: 시스템은 노션의 활동 콘텐츠를 자동으로 동기화할 수 있다.

FR7: 운영자는 개발 지식 없이도 활동 콘텐츠를 업데이트할 수 있다.

FR8: 노션 수정 후 5~30분 이내에 사이트에 반영된다. (초기 MVP는 주기적 동기화, 향후 서버리스 프록시로 거의 실시간 확장 가능)

FR9: 동기화 실패 시 사용자에게 오류 메시지와 마지막 업데이트 시각을 표시하며, 페이지는 깨지지 않는다.

FR10: 운영자는 노션 템플릿에 따라 필수 필드(제목, 날짜, 요약, 본문, 슬러그 또는 ID, 공개 여부)를 작성할 수 있으며, 필수 필드가 없을 경우 시스템이 적절히 처리한다.

FR11: 활동 콘텐츠는 검색 엔진에 노출될 수 있다.

FR12: 각 활동은 고유한 URL을 가진다.

FR13: 각 활동 상세 페이지는 메타 태그(title, description)와 Open Graph 태그(OG:title, OG:description, OG:image 등)를 제공한다.

FR14: 사용자는 키보드만으로 모든 기능에 접근할 수 있다.

FR15: 사용자는 모바일 기기에서 활동 페이지를 사용할 수 있다.

FR16: 사용자는 스크린 리더를 통해 활동 콘텐츠에 접근할 수 있다.

FR17: 외부 데이터 소스(노션) 장애 시에도 페이지가 완전히 깨지지 않고, 캐시된 데이터 또는 기본 메시지를 표시한다.

FR18: 시스템은 데이터 로드 실패 시 사용자에게 명확한 에러 메시지를 제공한다.

FR19: 사용자는 활동 페이지에서 다른 페이지(소개, 문의, SNS)로 자연스럽게 이동할 수 있다.

FR20: 사용자는 활동 상세 페이지의 외부 링크(SNS, 문의 페이지 등)를 클릭할 수 있다.

FR21: 시스템은 활동 데이터의 필수 필드(제목, 날짜, 요약, 본문)를 인식하고 처리할 수 있다.

FR22: 시스템은 선택적 필드(이미지, 첨부 자료, 카테고리 등)를 지원할 수 있다.

FR23: 시스템은 활동의 공개 여부 상태를 처리할 수 있다. (공개/비공개)

FR24: 활동 목록 페이지는 빠르게 로드된다. (정적 페이지 수준의 성능)

FR25: 활동 상세 페이지는 빠르게 로드된다.

FR26: 사용자는 활동 목록을 스크롤하여 더 많은 활동을 볼 수 있다. (필요 시 페이지네이션 또는 무한 스크롤)

FR27: 시스템은 활동의 날짜를 사용자가 이해하기 쉬운 형식으로 표시할 수 있다.

FR28: 시스템은 활동의 본문 내용을 읽기 쉬운 형식으로 표시할 수 있다. (마크다운 또는 HTML 렌더링)

FR29: 시스템은 활동 목록에서 최신 활동을 우선적으로 표시할 수 있다.

FR30: 운영자는 노션에서 활동을 추가한 후 사이트에 자동으로 반영되는 것을 확인할 수 있다.

FR31: 운영자는 노션 템플릿을 사용하여 일관된 형식으로 활동을 작성할 수 있다.

FR32: 운영자는 실수로 잘못된 형식의 데이터를 입력해도 사이트가 깨지지 않는다.

### NonFunctional Requirements

NFR1: Mobile(4G) 네트워크 환경에서 First Contentful Paint (FCP)는 1.5초 이하여야 한다.

NFR2: Mobile(4G) 네트워크 환경에서 Largest Contentful Paint (LCP)는 2.5초 이하여야 한다.

NFR3: Mobile(4G) 네트워크 환경에서 Time to Interactive (TTI)는 3.5초 이하여야 한다.

NFR4: 활동 목록 데이터 렌더링이 완료되는 시간은 캐시 미스 시 2.0초 이하여야 한다.

NFR5: 활동 목록 데이터 렌더링이 완료되는 시간은 캐시 히트 시 1.0초 이하여야 한다.

NFR6: 노션에서 활동 콘텐츠 수정 후 사이트에 반영되는 동기화 지연은 P95 기준 30분 이하여야 한다. (동기화 주기: 10~30분)

NFR7: 동기화 실패 시 이전 캐시된 데이터를 유지하여 사용자에게 표시해야 한다.

NFR8: 외부 데이터 소스(노션) 장애 시에도 페이지가 완전히 깨지지 않아야 한다.

NFR9: 데이터 로드 실패 시 사용자에게 오류 메시지와 마지막 업데이트 시각을 표시해야 한다.

NFR10: 가능한 경우 캐시된 데이터를 폴백으로 사용하여 콘텐츠를 표시해야 한다.

NFR11: 각 활동은 고유한 URL을 가져야 한다.

NFR12: 각 활동 상세 페이지는 메타 태그(title, description)와 Open Graph 태그(OG:title, OG:description, OG:image 등)를 포함해야 한다.

NFR13: 가능한 경우 활동 상세 페이지는 정적 HTML로도 접근 가능하도록 배포 시 생성되거나 프리렌더링되어야 한다. (검색 엔진 크롤링을 위해)

NFR14: 웹사이트는 WCAG 2.1 AA 수준을 목표로 하며, 최소한 A 수준을 준수해야 한다.

NFR15: 모든 기능은 키보드 네비게이션과 포커스 관리로 접근 가능해야 한다.

NFR16: 320px 너비의 모바일 화면에서도 모든 기능이 사용 가능해야 한다.

NFR17: 시스템은 민감한 데이터를 처리하지 않는다. (공개 활동 기록만 다룸)

NFR18: 노션 API 키 또는 토큰은 클라이언트 사이드 코드에 포함되지 않아야 한다. 서버 사이드 또는 GitHub Actions에서만 사용되어야 한다.

NFR19: 시스템은 일일 50~500명의 기본 사용자를 지원할 수 있어야 한다.

NFR20: 이벤트 시 일일 2,000~10,000명의 스파이크 트래픽을 캐싱 전략으로 대응할 수 있어야 한다.

### Additional Requirements

- **정적 사이트 아키텍처 유지**: 기존 정적 웹사이트 구조를 유지하면서 활동 공유 페이지에만 동적 기능 추가
- **GitHub Pages 호스팅**: GitHub Pages 환경에서 동작해야 하며, 정적 호스팅 제약 내에서 동적 기능 구현
- **노션 연동**: 노션을 콘텐츠 관리 도구로 사용하며, 노션 API 또는 Export 기능을 통해 데이터 연동
- **자동 동기화**: GitHub Actions를 통한 주기적 업데이트 (5~30분 간격) 또는 빌드 타임에 노션 데이터를 정적 파일로 생성
- **데이터 동기화 전략**: 노션 데이터를 JSON 또는 Markdown으로 Export하여 빌드 타임에 정적 파일 생성, 또는 클라이언트 사이드에서 API 호출 (캐싱 전략 필수)
- **에러 처리**: 외부 데이터 소스 장애 시에도 페이지가 깨지지 않도록 폴백 메커니즘 구현
- **확장성 고려**: 향후 백엔드 또는 CMS로 자연스럽게 확장 가능한 구조 설계
- **SEO 최적화**: 가능하면 빌드 타임에 노션 데이터를 정적 HTML로 생성하여 검색 엔진 노출 최적화
- **반응형 디자인**: 모바일/태블릿/데스크톱 지원, 터치 타겟 크기 준수 (최소 44x44px)
- **브라우저 지원**: 모던 브라우저 (Chrome, Edge, Firefox, Safari 최신 2버전, iOS Safari 15+, Android Chrome 최신 2버전)
- **성능 최적화**: 이미지 최적화 및 지연 로딩, CSS/JavaScript 모듈화 및 최소화, 외부 데이터 소스 캐싱 전략
- **접근성**: 시맨틱 HTML 사용, ARIA 레이블 사용, 키보드 네비게이션 지원
- **보안**: 노션 API 키는 GitHub Actions 또는 서버 사이드에서만 사용, 클라이언트 사이드에 노출 금지

**FRs covered:** FR1, FR2, FR3, FR4, FR11, FR12, FR13, FR19, FR20, FR21, FR22, FR23, FR24, FR25, FR26, FR27, FR28, FR29

### Epic 2: 노션 연동 및 자동 동기화

운영자가 노션에서 활동 콘텐츠를 작성하고 수정하면 자동으로 사이트에 반영되도록 한다. 개발 지식 없이도 콘텐츠를 업데이트할 수 있으며, 외부 데이터 소스 장애 시에도 페이지가 안정적으로 동작하도록 에러 처리 및 폴백 메커니즘을 구현한다.

**FRs covered:** FR5, FR6, FR7, FR8, FR9, FR10, FR17, FR18, FR30, FR31, FR32

### Epic 3: 접근성 및 사용성

모든 사용자가 키보드, 모바일 기기, 스크린 리더를 통해 활동 페이지에 접근할 수 있도록 한다. WCAG 2.1 기준을 준수하여 포용적인 사용자 경험을 제공한다.

**FRs covered:** FR14, FR15, FR16

## Epic 1: 활동 콘텐츠 표시 및 탐색

외부 방문자가 활동 목록을 시간순으로 보고, 관심 있는 활동의 상세 내용을 확인할 수 있도록 한다. 검색 엔진 최적화를 포함하여 활동 콘텐츠가 검색 가능하도록 하고, 각 활동은 고유한 URL과 메타 태그를 가진다.

### Story 1.1: 활동 데이터 구조 정의 및 기본 파싱

As a 시스템,
I want 활동 데이터의 필수 필드와 선택적 필드를 인식하고 파싱할 수 있도록,
So that 활동 콘텐츠를 일관되게 처리하고 표시할 수 있다.

**Acceptance Criteria:**

**Given** 노션에서 가져온 활동 데이터가 있다
**When** 시스템이 활동 데이터를 파싱한다
**Then** 필수 필드(제목, 날짜, 요약, 본문)를 인식하고 추출한다
**And** 선택적 필드(이미지, 첨부 자료, 카테고리 등)가 있으면 추출한다
**And** 활동의 공개 여부 상태를 인식하고 처리한다
**And** 필수 필드가 누락된 경우 적절한 기본값 또는 에러 처리를 한다

### Story 1.2: 활동 목록 페이지 구현

As a 외부 방문자,
I want 활동 목록을 시간순으로 보고 각 활동의 기본 정보를 확인할 수 있도록,
So that 최신 활동을 빠르게 파악하고 관심 있는 활동을 찾을 수 있다.

**Acceptance Criteria:**

**Given** 활동 데이터가 파싱되어 있다
**When** 사용자가 활동 목록 페이지에 접근한다
**Then** 활동들이 시간순(최신 우선)으로 정렬되어 표시된다
**And** 각 활동에 제목, 날짜, 간단 설명이 표시된다
**And** 날짜는 사용자가 이해하기 쉬운 형식으로 표시된다 (예: "2026년 1월 12일")
**And** 페이지는 빠르게 로드된다 (정적 페이지 수준의 성능, NFR4, NFR5)
**And** 사용자는 스크롤하여 더 많은 활동을 볼 수 있다
**And** 공개된 활동만 표시된다 (비공개 활동은 제외)

### Story 1.3: 활동 상세 페이지 구현

As a 외부 방문자,
I want 관심 있는 활동의 상세 내용을 확인할 수 있도록,
So that 활동의 구체적인 내용과 세부사항을 이해할 수 있다.

**Acceptance Criteria:**

**Given** 활동 목록이 표시되어 있다
**When** 사용자가 특정 활동을 클릭한다
**Then** 해당 활동의 상세 내용이 표시된다
**And** 본문 내용이 읽기 쉬운 형식으로 렌더링된다 (마크다운 또는 HTML)
**And** 페이지는 빠르게 로드된다 (NFR4, NFR5)
**And** 공개된 활동만 상세 내용을 볼 수 있다

### Story 1.4: 고유 URL 및 라우팅

As a 외부 방문자,
I want 각 활동에 고유한 URL을 통해 접근할 수 있도록,
So that 특정 활동을 직접 링크로 공유하거나 북마크할 수 있다.

**Acceptance Criteria:**

**Given** 활동 데이터에 슬러그 또는 ID가 있다
**When** 시스템이 활동 상세 페이지를 생성한다
**Then** 각 활동은 고유한 URL을 가진다 (예: /activities/{slug} 또는 /activities/{id})
**And** URL을 통해 직접 접근할 수 있다
**And** 브라우저 뒤로가기/앞으로가기가 정상 작동한다
**And** 잘못된 URL 접근 시 적절한 에러 페이지 또는 리다이렉트를 표시한다

### Story 1.5: SEO 최적화 (메타 태그 및 정적 생성) - MVP 범위

As a 검색 엔진,
I want 활동 콘텐츠가 검색 가능하고 메타 정보를 제공하도록,
So that 활동 페이지가 검색 결과에 노출되고 사용자가 찾을 수 있다.

**Acceptance Criteria:**

**Given** 활동 상세 페이지가 있다
**When** 페이지가 로드된다
**Then** 각 활동 상세 페이지는 메타 태그(title, description)를 포함한다
**And** Open Graph 태그(OG:title, OG:description)를 포함한다
**And** 가능한 경우 활동 상세 페이지는 정적 HTML로도 접근 가능하다 (빌드 타임 생성 또는 프리렌더링)
**And** 검색 엔진 크롤러가 콘텐츠를 인덱싱할 수 있다
**And** MVP 범위 내에서 기본적인 SEO 최적화가 구현된다

### Story 1.6: 네비게이션 및 외부 링크

As a 외부 방문자,
I want 활동 페이지에서 다른 페이지나 외부 링크로 이동할 수 있도록,
So that 단체의 다른 정보를 확인하거나 추가 행동을 취할 수 있다.

**Acceptance Criteria:**

**Given** 활동 목록 또는 상세 페이지가 표시되어 있다
**When** 사용자가 네비게이션 메뉴를 클릭한다
**Then** 다른 페이지(소개, 문의, SNS 등)로 자연스럽게 이동할 수 있다
**And** 활동 상세 페이지에 외부 링크(SNS, 문의 페이지 등)가 있다면 클릭 가능하다
**And** 링크는 새 탭에서 열리거나 적절한 방식으로 동작한다
**And** 기존 사이트의 네비게이션 구조와 일관성을 유지한다

## Epic 2: 노션 연동 및 자동 동기화

운영자가 노션에서 활동 콘텐츠를 작성하고 수정하면 자동으로 사이트에 반영되도록 한다. 개발 지식 없이도 콘텐츠를 업데이트할 수 있으며, 외부 데이터 소스 장애 시에도 페이지가 안정적으로 동작하도록 에러 처리 및 폴백 메커니즘을 구현한다.

### Story 2.1: 노션 데이터 소스 연결 및 인증 설정

As a 시스템,
I want 노션 데이터 소스에 안전하게 연결하고 인증할 수 있도록,
So that 운영자가 노션에서 작성한 콘텐츠를 가져올 수 있다.

**Acceptance Criteria:**

**Given** 노션 데이터베이스 또는 페이지가 설정되어 있다
**When** 시스템이 노션 데이터 소스에 연결한다
**Then** 노션 API 또는 Export 기능을 통해 데이터에 접근할 수 있다
**And** 노션 API 키 또는 토큰은 GitHub Actions 또는 서버 사이드에서만 사용된다 (NFR18)
**And** 클라이언트 사이드 코드에 인증 정보가 노출되지 않는다
**And** 운영자는 개발 지식 없이도 노션에서 활동 콘텐츠를 작성하고 수정할 수 있다 (FR7)
**And** 연결 실패 시 적절한 에러 로깅이 이루어진다

### Story 2.2: 노션 데이터 가져오기 및 변환

As a 시스템,
I want 노션에서 활동 데이터를 가져와서 표준 형식으로 변환할 수 있도록,
So that 활동 콘텐츠를 일관되게 처리하고 표시할 수 있다.

**Acceptance Criteria:**

**Given** 노션 데이터 소스에 연결되어 있다
**When** 시스템이 노션에서 활동 데이터를 가져온다
**Then** 노션 템플릿에 따라 필수 필드(제목, 날짜, 요약, 본문, 슬러그 또는 ID, 공개 여부)를 추출한다 (FR10)
**And** 선택적 필드(이미지, 첨부 자료, 카테고리 등)가 있으면 추출한다
**And** 필수 필드가 누락된 경우 적절한 기본값 또는 에러 처리를 한다 (FR10)
**And** 활동 데이터의 필수 필드를 인식하고 처리할 수 있다 (FR21)
**And** 선택적 필드를 지원할 수 있다 (FR22)
**And** 활동의 공개 여부 상태를 처리할 수 있다 (FR23)
**And** 노션 템플릿 형식에 맞지 않는 데이터도 안전하게 처리한다

### Story 2.3: 자동 동기화 메커니즘 구현

As a 운영자,
I want 노션에서 활동을 추가하거나 수정하면 자동으로 사이트에 반영되도록,
So that 개발 지식 없이도 콘텐츠를 업데이트할 수 있다.

**Acceptance Criteria:**

**Given** 노션 데이터 가져오기 및 변환이 구현되어 있다
**When** GitHub Actions가 주기적으로 실행된다 (5~30분 간격)
**Then** 노션에서 최신 활동 데이터를 가져온다 (FR6)
**And** 변경된 내용을 감지하고 사이트에 반영한다
**And** 노션 수정 후 5~30분 이내에 사이트에 반영된다 (FR8, NFR6)
**And** 동기화가 완료되면 자동으로 사이트가 배포된다
**And** 운영자는 노션에서 활동을 추가한 후 사이트에 자동으로 반영되는 것을 확인할 수 있다 (FR30)
**And** 동기화 프로세스는 로그에 기록된다

### Story 2.4: 에러 처리 및 폴백 메커니즘

As a 사용자,
I want 외부 데이터 소스 장애 시에도 페이지가 안정적으로 동작하도록,
So that 서비스 중단 없이 활동 콘텐츠를 확인할 수 있다.

**Acceptance Criteria:**

**Given** 외부 데이터 소스(노션)에 장애가 발생했다
**When** 시스템이 데이터를 로드하려고 시도한다
**Then** 페이지가 완전히 깨지지 않고, 캐시된 데이터 또는 기본 메시지를 표시한다 (FR17, NFR8)
**And** 동기화 실패 시 사용자에게 오류 메시지와 마지막 업데이트 시각을 표시한다 (FR9, NFR9)
**And** 가능한 경우 캐시된 데이터를 폴백으로 사용하여 콘텐츠를 표시한다 (NFR7, NFR10)
**And** 데이터 로드 실패 시 사용자에게 명확한 에러 메시지를 제공한다 (FR18)
**And** 운영자가 실수로 잘못된 형식의 데이터를 입력해도 사이트가 깨지지 않는다 (FR32)
**And** 에러는 로그에 기록되고 모니터링할 수 있다

### Story 2.5: 운영자 경험 개선

As a 운영자,
I want 노션 템플릿을 사용하여 일관된 형식으로 활동을 작성할 수 있도록,
So that 실수 없이 안정적으로 콘텐츠를 관리할 수 있다.

**Acceptance Criteria:**

**Given** 운영자가 노션에서 활동을 작성한다
**When** 노션 템플릿을 사용한다
**Then** 일관된 형식으로 활동을 작성할 수 있다 (FR31)
**And** 템플릿에는 필수 필드와 선택적 필드가 명확히 정의되어 있다
**And** 필수 필드가 누락된 경우 시스템이 적절히 처리한다 (FR10)
**And** 잘못된 형식의 데이터를 입력해도 사이트가 깨지지 않는다 (FR32)
**And** 운영자는 템플릿 사용 가이드를 참고할 수 있다
**And** 템플릿 예시가 제공된다

## Epic 3: 접근성 및 사용성

모든 사용자가 키보드, 모바일 기기, 스크린 리더를 통해 활동 페이지에 접근할 수 있도록 한다. WCAG 2.1 기준을 준수하여 포용적인 사용자 경험을 제공한다.

### Story 3.1: 키보드 접근성 구현

As a 키보드 사용자,
I want 키보드만으로 모든 기능에 접근할 수 있도록,
So that 마우스 없이도 활동 페이지를 완전히 사용할 수 있다.

**Acceptance Criteria:**

**Given** 활동 목록 또는 상세 페이지가 표시되어 있다
**When** 사용자가 키보드만을 사용하여 탐색한다
**Then** 모든 인터랙티브 요소(링크, 버튼, 입력 필드 등)에 키보드로 접근할 수 있다 (FR14, NFR15)
**And** Tab 키로 포커스를 이동할 수 있고, Shift+Tab으로 역방향 이동이 가능하다
**And** Enter 또는 Space 키로 요소를 활성화할 수 있다
**And** 포커스가 시각적으로 명확하게 표시된다 (포커스 링 또는 하이라이트)
**And** 포커스 순서가 논리적이고 직관적이다
**And** 키보드 트랩이 발생하지 않는다 (모든 요소에서 탈출 가능)
**And** 활동 목록과 상세 페이지 모두 키보드로 완전히 탐색 가능하다

### Story 3.2: 모바일 반응형 디자인 구현

As a 모바일 사용자,
I want 모바일 기기에서 활동 페이지를 편리하게 사용할 수 있도록,
So that 어떤 디바이스에서도 최적의 경험을 얻을 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 모바일 기기(스마트폰, 태블릿)에서 활동 페이지에 접근한다
**When** 페이지가 로드된다
**Then** 320px 너비의 모바일 화면에서도 모든 기능이 사용 가능하다 (FR15, NFR16)
**And** 콘텐츠가 화면 크기에 맞게 적절히 조정된다 (반응형 레이아웃)
**And** 터치 타겟 크기가 최소 44x44px을 준수한다
**And** 텍스트가 읽기 쉬운 크기로 표시된다
**And** 스크롤이 부드럽고 자연스럽게 작동한다
**And** 모바일, 태블릿, 데스크톱에서 모두 최적화된 레이아웃이 제공된다
**And** 뷰포트 메타 태그가 올바르게 설정되어 있다

### Story 3.3: 스크린 리더 접근성 구현

As a 스크린 리더 사용자,
I want 스크린 리더를 통해 활동 콘텐츠에 접근할 수 있도록,
So that 시각 장애가 있어도 활동 정보를 완전히 이해할 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 스크린 리더를 사용하여 활동 페이지에 접근한다
**When** 페이지가 로드된다
**Then** 시맨틱 HTML 요소가 올바르게 사용되어 콘텐츠 구조가 명확하다 (FR16, NFR14)
**And** ARIA 레이블과 역할 속성이 적절히 사용되어 요소의 목적이 명확하다
**And** 이미지에 대체 텍스트(alt)가 제공된다
**And** 링크와 버튼의 목적이 텍스트로 명확히 전달된다
**And** 활동 목록과 상세 내용이 스크린 리더로 읽을 수 있다
**And** 웹사이트는 최소한 WCAG 2.1 A 수준을 준수한다 (NFR14)
**And** 스크린 리더 사용자가 모든 기능을 독립적으로 사용할 수 있다
**And** 포커스된 요소의 상태와 콘텐츠가 스크린 리더로 정확히 전달된다
